# Tesla Supercharger Access

Agent-facing notes for this repo.

The canonical human-facing docs are:

- `README.md`
- `docs/ARCHITECTURE.md`

Read those first before changing the pipeline.

## Project Goal

Build an interactive map answering:

> What fraction of the world's population lives within X km of an open Tesla Supercharger?

The current working implementation is a static frontend with precomputed coverage tiles and country-level analytics. Older backend/deck.gl code may still exist, but it is not the current app path.

## Current Working Architecture

- Frontend: `frontend/index.html`
- Map library: MapLibre GL JS
- Coverage layer: MapLibre custom WebGL layer
- Coverage data: PNG tile pyramid under `frontend/tiles/{z}/{x}/{y}.png`
- Global stats: `frontend/pop_cumulative.json`
- Road-distance A/B outputs:
  - `data/pixel_distances_road.npz`
  - `frontend/tiles_road/`
  - `frontend/pop_cumulative_road.json`
  - `frontend/country_stats_road.json`
- Country stats:
  - `frontend/countries.geojson`
  - `frontend/country_stats.json`
- Viewport stats:
  - `frontend/viewport_manifest.json`
  - `frontend/viewport_pop_total.bin`
  - `frontend/viewport_pop_covered.bin`
  - These are still generated, but the current UI no longer displays viewport stats.

The browser thresholds precomputed distance tiles in the shader. Slider updates are instant because no new data is fetched for each radius.

## Important Current Semantics

Coverage tiles use RG16 distance encoding:

```text
R/G = 16-bit distance code
B   = reserved
A   = log-scaled population alpha
```

Distance encoding:

```text
code = ceil(clamp(distance_km / 500, 0, 1) * 65535)
```

Radius threshold:

```text
radius_code = ceil(radius_km / 500 * 65535)
covered if code <= radius_code
```

This is deliberate. Do not revert to 8-bit red-channel distance encoding. The old 8-bit version made `0 km` wrong by collapsing all distances below about 2 km to zero.

Stats also use ceil semantics:

```text
bucket = ceil(distance_km)
```

So `0 km` includes only true zero-distance cells.

## Data Pipeline

Run from project root:

```powershell
.venv/Scripts/python scripts/extract_pixels.py
.venv/Scripts/python scripts/extract_chargers.py
.venv/Scripts/python scripts/compute_distances.py
$env:TILE_ENCODE_WORKERS = "16"
.venv/Scripts/python scripts/precompute_tiles.py 11
.venv/Scripts/python scripts/build_outputs.py
.venv/Scripts/python scripts/precompute_viewport_stats.py
.venv/Scripts/python scripts/precompute_country_stats.py
.venv/Scripts/python scripts/validate_outputs.py
```

Road-distance merge and side-by-side rebuild:

```powershell
.venv/Scripts/python scripts/merge_road_distances.py
.venv/Scripts/python scripts/precompute_road_region_tiles.py --region DEU --workers 1 --max-pending 2
.venv/Scripts/python scripts/build_outputs.py --source road
.venv/Scripts/python scripts/precompute_viewport_stats.py --source road
.venv/Scripts/python scripts/precompute_country_stats.py --source road
.venv/Scripts/python scripts/validate_outputs.py --source road
```

`DISTANCE_SOURCE=road` is equivalent to passing `--source road`.

Important: do not use `scripts/precompute_tiles.py 11 --source road` for the current Germany-only road dataset on the 16 GB laptop. That path attempts a full global road tile pyramid. Use `scripts/precompute_road_region_tiles.py` instead; it writes sparse regional override tiles and the frontend falls back to `frontend/tiles/` for non-road pixels.

For tile/frontend/stat format changes, usually only this subset is needed:

```powershell
$env:TILE_ENCODE_WORKERS = "16"
.venv/Scripts/python scripts/precompute_tiles.py 11
.venv/Scripts/python scripts/build_outputs.py
.venv/Scripts/python scripts/precompute_viewport_stats.py
.venv/Scripts/python scripts/precompute_country_stats.py
.venv/Scripts/python scripts/validate_outputs.py
```

`precompute_tiles.py` reuses geometry caches in `data/npy/`.

## Country Stats

Country stats are generated by:

```powershell
.venv/Scripts/python scripts/precompute_country_stats.py
```

Default country boundary source:

```text
Natural Earth Admin 0 Countries 10m
```

Country assignment method:

- population cells are assigned by source-cell center,
- chargers are assigned by point location,
- both use bounded temporary raster windows, not a full global country raster.

Do not reintroduce a full global 100 m country-id raster cache. It created a roughly 130 GB file and crashed the machine.

Manual charger overrides are read from:

```text
data/country_charger_overrides.csv
```

Expected columns:

```csv
charger_index,iso_a3,country_name
```

Review helper:

```powershell
.venv/Scripts/python scripts/export_unassigned_chargers.py
```

Review page:

```text
frontend/review_unassigned_chargers.html
```

Current charger assignment status after overrides:

```text
Assigned chargers: 7,854 / 7,854
```

Current population assignment status:

```text
Assigned country population: 8,404,231,351 / 8,480,668,252
```

Unassigned population review files:

```text
frontend/unassigned_population_bins.csv
frontend/unassigned_population_bins.geojson
```

These are review outputs only; they are not yet applied to country stats.

Important semantic caveat: Natural Earth sovereign country features may include overseas territories. France is the known example; it may not reach 100% at 500 km because the `France` feature includes overseas territories.

## Tile Geometry

Do not approximate the source cells as lat/lon rectangles.

The correct path is:

1. Keep `x_moll` and `y_moll` from the original GHS-POP Mollweide raster.
2. Build exact 100 m source-cell corners in Mollweide.
3. Project those corners to WebMercator.
4. Rasterize the projected footprint into z11 WebMercator tile pixels.
5. Build lower zooms bottom-up from the z11 rasterization.

Relevant caches:

```text
data/npy/wm_z11_x0.npy
data/npy/wm_z11_x1.npy
data/npy/wm_z11_y0.npy
data/npy/wm_z11_y1.npy
data/npy/wm_z11_row_offsets.npy
data/npy/wm_z11_row_indices.npy
```

These caches are required for the clean aligned grid.

## Frontend Display Modes

The frontend has:

- `Exact`: raw raster cells.
- `Auto`: shader-side smoothing at lower/mid zoom and non-tiny radii.
- `Distance`: Great-circle or Road (Germany only for current computed data).

Smoothing is display-only. It must not affect stats or tile data.

Road distance is computed per OSM region. Germany is currently present. Infinite or missing road cells intentionally fall back to great-circle during `scripts/merge_road_distances.py`.

## Current UI

`frontend/index.html` currently has:

- interaction panel:
  - search box,
  - country selector,
  - distance slider plus editable red distance value,
  - Auto/Cells display mode,
- results panel:
  - selected-country population access,
  - selected-country Supercharger count and density,
  - world population access,
- compact legend panel in the bottom-right.

Avoid vague UI labels like "coverage" where possible. Prefer explicit wording such as:

```text
X% of France's population lives within 50 km of Tesla's Supercharger network.
```

## Validation Checklist

After a rebuild:

```powershell
.venv/Scripts/python scripts/validate_outputs.py
```

Manual checks:

- `0 km` should not show broad green blobs.
- `1 km`, `2 km`, and `5 km` should grow gradually.
- `Exact` mode should show aligned source-cell footprints.
- `Auto` mode should smooth overview zooms without changing high-zoom truth.
- Tile boundaries should not show seams or duplicate strips.

## Known Limitations

- Distance is from source-cell center to charger coordinate, not from every point inside the cell.
- Viewport stats are approximate 1 degree bin sums.
- Native tile max is currently z11. Zooming beyond z11 magnifies z11 textures.
- The source population raster is 100 m. Smoothing can improve overview appearance, but it cannot create sub-cell detail.
